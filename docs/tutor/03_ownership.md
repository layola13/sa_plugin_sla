# 所有权与生命周期

Sla 的核心安全特征是其静态所有权与借用分析机制。它不依赖垃圾回收，而是通过严格编译期规则保障内存安全。

---

## 1. 移动语义与移动操作符 `^`

在 Sla 中，管理堆资源、FFI 句柄或没有实现显式复制（Copy）的复合资源类型，属于**拥有权类型**。
* **所有权转移**：当一个拥有权类型被赋值给另一个变量，或者作为参数传递给非借用函数时，该资源的所有权会被 **Move（移动）**。
* **移动操作符 `^`**：对于拥有权类型，在转移所有权时，必须在变量名前加 `^` 符号。这向开发者明确指示了资源的转移点。
  ```sla
  let my_box = Box::new(500);
  let new_box = ^my_box; // 所有权显式移动到 new_box
  // 此时 my_box 在编译器符号表中被标记为 Consumed（已消费）
  ```
* **Use-After-Move 保护**：若在此之后尝试再次读取已转移所有权的变量，Sla 编译器会产生编译错误：
  ```sla
  // let val = my_box.value; // 编译错误！检测到 UseAfterMove 
  ```

---

## 2. 借用语义与借用操作符 `&`

为了避免频繁移动所有权以及不需要转移所有权时的资源共享，Sla 提供了引用借用机制。
* **借用操作符 `&`**：使用 `&` 创建指向原变量的借用指针。
  ```sla
  let value = 100;
  let ref_val = &value; // ref_val 是对 value 的只读借用
  ```
* **函数参数借用**：在函数定义中使用 `&` 修饰参数类型或 `&self`，这要求调用方传入一个借用视图（前缀带有 `&`）。
  ```sla
  fn print_value(val: &int) {
      // 仅读取值
  }
  
  let x = 42;
  print_value(&x); // 传入借用指针，x 并没有发生移动
  ```
* **生命周期锁定 (Locked_Read / Locked_Mut)**：借用检查器（Borrow Checker）会在编译期分析变量的生命周期，确保在借用持续期间，原所有权变量不能被移动或以冲突的方式修改。

---

## 3. 自动资源释放 (Implicit Cleanup)

在底层的 SA 汇编中，每个分配的资源以及每个活跃的引用最终都需要手写 `!reg` 进行清理或解锁。在 Sla 中，这一繁重的体力劳动由编译器完全自动化：

* **退出作用域自动释放**：当变量（无论是值类型还是借用视图）离开其定义的代码块时，编译器会自动在降低为 SA 时生成 `!reg` 的释放代码，防止发生内存泄漏（Memory Leak）。
* **分支合并（Phi 自动平衡）**：
  若在 `if/else` 的其中一个分支中消耗了某个变量（例如通过 `^` 移动给别的函数），而另一个分支中没有消耗，那么在控制流合并点上会出现活跃状态冲突。Sla 编译器能自动识别此不平衡性，在没有消耗的分支末尾自动补全相应的释放操作，消除 Phi 冲突：
  ```sla
  let resource = Box::new(99);
  if cond {
      consume_resource(^resource); // resource 在此分支被移动消费
  } else {
      // 编译器在此处自动生成 !resource 以便释放资源
  }
  // 合并点后，resource 在两条路径上都确定处于 Consumed 状态
  ```

* **关于析构函数**：Sla 秉持最小化原则，**没有**定义任何自定义析构函数机制，亦不支持手写 `drop` 关键字或 `drop()` 函数。若变量是外部资源类型（如由外部 `.sal` 声明且绑定了析构器的智能指针 `Rc`），当其生命周期结束时，Sla 编译器会自动调用注册的释放 FFI 方法（如 `sa_std_rc_release`），以此来妥善清理资源。\n
